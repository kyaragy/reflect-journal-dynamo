# Codex向け実装指示書：思考イベントカード機能改修

## 0. 前提
この指示書は、既存の Reflect Journal アプリに対して、思考イベントカード機能の改修を実施するためのものです。  
要件の詳細は、同じ作業領域に配置済みの要件定義Markdownを参照してください。

参照先ファイル名:
- `reflect_journal_thought_event_requirements.md`

本指示書は、Codex が実装・設計・テストを進めるための実務指示をまとめたものです。  
要件定義書が正とし、本指示書は実装時の優先順位・進め方・期待成果物を補足するものとします。

---

## 1. 今回の改修目的
既存のジャーナル記録機能を、以下の目的により拡張してください。

- 1カードを、1つの思考イベントのまとまりとして扱えるようにする
- 既存のカテゴリ（tag）を、任意の補助分類として維持する
- カードに「きっかけ」を持たせる
- カード内に、思考・感情・行動・身体反応の時系列ステップを複数持てるようにする
- 後続の振り返りやChatGPTとの対話に使いやすいMarkdown出力を実現する
- スマホでは入力しやすく、PCでは一覧性と編集効率を高めたUIにする
- 見た目としても、既存より整理され、クリーンでモダンな印象に改善する

---

## 2. 最重要の実装方針
以下は特に重要です。実装時に必ず守ってください。

### 2-1. 先に設計してから実装すること
いきなり実装に入らず、まず以下を整理してください。

- データ構造
- 型定義
- 画面状態
- コンポーネント分割方針
- 既存データとの整合方針
- Markdown出力構造

必要であれば、リポジトリ内に設計メモを残してください。

### 2-2. 要件定義書を優先すること
不明点があれば、まず `reflect_journal_thought_event_requirements.md` を確認してください。  
実装判断が必要な場合は、以下の優先順位で解釈してください。

1. 要件定義書の明記内容
2. 要件定義書のコンセプトと重要方針
3. 既存実装との整合
4. MVPとしての簡潔さ

### 2-3. MVPの範囲を守ること
今回の改修では、欲張って高度機能を入れないでください。  
特に以下はMVP対象外です。

- ステップ並び替え
- ステップ途中挿入
- 複数きっかけ
- 自動分類
- 高度なアニメーション
- 複雑なショートカット最適化

## 3. 実装対象

### 3-1. データ構造の改修
思考イベントカードとして扱えるよう、カードデータを再設計してください。

最低限、以下を表現できるようにしてください。

- カードID
- 作成日時 / 更新日時
- カテゴリ（任意、単一選択）
- きっかけ
  - 種別
  - 内容
- ステップ配列
  - 順序
  - 種別
  - 内容

#### きっかけ種別
- `external`
- `internal`
- `physical`

#### カテゴリ
- `work`
- `private`
- `study`

#### ステップ種別
- `thought`
- `emotion`
- `action`
- `body`

型は曖昧な string 乱用ではなく、TypeScript の union type または enum 相当で安全に扱ってください。

### 3-2. 既存データとの互換
既存のジャーナルデータがある場合、極力壊さないでください。  
ただし完全な後方互換が難しい場合は、以下のいずれかを選んでください。

- 移行処理を設ける
- 旧データを読み取り専用として扱う
- 新旧を判別できる構造にする

どれを採るかは、実装コストと安全性を踏まえて最も妥当な案を選び、その理由をPR説明に明記してください。

### 3-3. カード入力UI
新規作成・編集画面では、以下を実現してください。

- カード上部に「きっかけ」を配置
- カード上部付近にカテゴリ選択UIを配置
- きっかけは種別選択 + テキスト入力で構成
- その下に時系列ステップを縦に並べる
- ステップは複数追加できる
- 各ステップは、種別選択 + 内容入力で構成
- ステップ削除ができる
- ステップはMVPでは末尾追加のみ

### 3-4. マークダウン出力
カード内容を、要件定義書に沿った構造でMarkdown出力できるようにしてください。

最低限、以下を含めてください。

- 作成日時
- カテゴリ（設定されている場合）
- きっかけ種別
- きっかけ内容
- ステップ一覧（時系列順）
- 各ステップの種別と内容
- ステップは `1.` の番号の下に `- 種別:` と `- 内容:` を持つ構造とすること

出力文言は自然な日本語にしてください。  
内部コード値（`external` 等）をそのまま出さず、ユーザー向けラベルへ変換してください。

## 4. UI/UXの指示

## 4-1. スマホ優先で入力体験を改善すること
今回の主要課題の1つはスマホ入力です。  
以下を重視してください。

- テキストエリアは入力内容に応じて縦に伸びること
- 一定以上は最大高さで制御すること
- キーボード表示中も操作しやすいこと
- 操作導線を過度に上部へ寄せないこと
- 外タップしないと完了できないUIにしないこと

### 4-2. PCでは情報密度を上げてよい
PC表示では、スマホの単純拡大版にしないでください。  
必要に応じて以下を検討してください。

- 2カラム構成
- きっかけ欄とステップ編集欄の分離
- 横幅を活かしたラベル配置
- より見渡しやすい余白設計

### 4-3. ステップ追加UIはシンプルにすること
見た目が雑多になるのを避けてください。  
MVPでは、以下の思想で実装してください。

- 追加導線は原則1つ
- 追加後に種別を選ぶ
- 種別ボタンを大量に並べない
- ドラッグ並び替えは入れない

## 5. デザイン実装の指示
要件定義書のデザイン要件を必ず反映してください。  
特に以下は明確に守ってください。

### 5-1. 使用技術
- 既存の Tailwind CSS ベースで実装すること
- アイコンは `lucide-react` を使うこと
- 新しい巨大UIライブラリは原則追加しないこと

### 5-2. 見た目の方向性
- クリーン
- ミニマル
- モダン
- 情報の階層が明確
- 入力の邪魔をしない

### 5-3. 具体的なデザインルール
- ベース背景は明るいニュートラルカラー
- カード背景は白系
- 主アクセントは青系または青緑系の落ち着いた色
- ステップ種別はチップやラベルで控えめに色分け
- カード、入力欄、ボタンの角丸ルールを統一
- シャドウは薄く、強すぎないこと
- 境界線は細く控えめにすること
- 情報が詰まりすぎないよう余白を確保すること

### 5-4. アイコン運用
以下のような意味単位でアイコンを付与してよいです。

- きっかけ
- 思考
- 感情
- 行動
- 身体反応
- 追加
- 保存
- 完了
- 削除
- Markdown出力

ただし、アイコンだけで意味を伝えず、ラベルまたは補助テキストと併用してください。

---

## 6. コンポーネント分割の期待
実装時は、責務が分かれるようにコンポーネントを分割してください。  
命名は既存プロジェクトの流儀に合わせて構いませんが、少なくとも以下の粒度は検討してください。

- 思考イベントカード全体
- きっかけ入力ブロック
- ステップ一覧
- ステップ行
- 種別選択UI
- Markdown出力関連処理

また、UIコンポーネントとデータ変換ロジックを不必要に密結合させないでください。

---

## 7. 状態管理・ロジックの指示
- 画面状態はできるだけ単純に保つこと
- フォーム状態が複雑化しすぎないようにすること
- 一時入力状態と保存済み状態の責務を分けること
- Markdown出力ロジックはUIから分離すること
- 表示ラベル変換と内部値の変換処理は共通化すること

---

## 8. テスト実装の指示
単体テストは必ず実装してください。  
最低限、以下をテスト対象に含めてください。

### 8-1. データ/変換系テスト
- きっかけ種別の内部値→表示名変換
- カテゴリの内部値→表示名変換
- ステップ種別の内部値→表示名変換
- Markdown出力が期待構造になること
- ステップが時系列順で出力されること

### 8-2. UI/操作系テスト
- カード新規作成時にきっかけ入力欄が表示されること
- カード新規作成時にカテゴリ選択UIが表示されること
- ステップ追加で末尾に新規ステップが追加されること
- ステップ種別を変更できること
- ステップ削除ができること
- 入力済み内容が編集画面で表示されること

### 8-3. 回帰観点
- 既存の一覧表示や保存処理を壊していないこと
- 既存のMarkdown出力導線がある場合、それを壊していないこと

テストは、表面的なレンダリング確認だけでなく、主要ユースケースを担保する内容にしてください。

## 9. 実装手順の推奨順序
以下の順で進めてください。

1. 要件定義書を読む
2. 既存構造を調査する
3. データ構造と型を設計する
4. Markdown出力仕様を先に固める
5. UIコンポーネント分割方針を決める
6. カード作成・編集UIを実装する
7. レスポンシブ調整を行う
8. デザイン調整を行う
9. 単体テストを追加する
10. 最後に不要コード・重複コードを整理する
