# 思考イベントカード機能 要件定義書

## 1. 文書概要

### 1-1. 目的
本機能は、ユーザーが1つの思考イベントのまとまりをカード単位で記録し、後続の振り返りやChatGPTとの対話に使いやすい形で保持・出力できるようにすることを目的とする。

記録時は、感情や本音をあるがままに書けることを重視する。  
振り返り時は、きっかけ・思考・感情・行動・身体反応の流れを時系列で追え、偏りや極端さ、抜けている視点に気づきやすいことを重視する。

### 1-2. 背景
本アプリは単なる日記ではなく、以下の流れを支援することを主眼とする。

1. その時点の本音や感情を、加工せずに書き出す  
2. 少し時間を置いて、冷静な状態で見返す  
3. ChatGPTとの対話で、自分だけでは出にくい観点に触れる  
4. 自分の偏り・極端さ・抜けている視点に気づく  
5. 次回以降の思考や行動を少しずつ更新していく  

### 1-3. スコープ
本書は以下を対象とする。

- 思考イベントカード機能の要件
- きっかけおよび時系列ステップのデータ構造
- スマホ・PCそれぞれのUI要件
- デザイン要件
- マークダウン出力要件
- MVP範囲

本書は以下を対象外とする。

- AIによる自動分析ロジックの詳細設計
- バックエンド実装方式の詳細
- DB物理設計
- 画面遷移図やコンポーネント分割の詳細設計

## 2. コンセプト

### 2-1. 本機能が解決したいこと
本機能は、以下の課題を扱う。

- 本音をオブラートに包まずに書き出し、ストレスを溜め込みすぎないようにする
- 思考が過激化・極端化していく流れを後から追えるようにする
- 自分だけでは気づきにくい考え方の偏りや抜けに、振り返りを通して気づきやすくする
- ChatGPTとの対話に渡しやすい構造化記録を残す

### 2-2. 機能コンセプト
本機能は以下3段階を支える。

#### 書く
- 感情や本音を、あるがままに記録する
- 記録時点では過度に整理させない

#### 寝かせる
- 書いた直後の感情の熱を少し下げる
- その場で結論づけるのではなく、後から見返せる状態にする

#### 振り返る
- 自分で見返す
- ChatGPTとの対話で別角度から問い返してもらう
- 偏りや極端さを見つける
- 次回の思考や行動に活かす

### 2-3. 重要方針
- 記録時は「正しさ」より「率直さ」を優先する
- 振り返り時は「客観性」と「時系列の読み解きやすさ」を重視する
- 入力時に複雑な整理や分類を強制しすぎない
- 後から対話しやすい構造を優先する

## 3. 用語定義

| 用語 | 定義 |
|---|---|
| カード | ユーザーが1つのまとまりとして振り返りたい思考イベントの記録単位 |
| カテゴリ | カードを大まかな文脈で見分けやすくするための補助分類。カードの主構造とは独立した任意項目 |
| きっかけ | そのカードにおける主たる発端。外部出来事、内部発生、身体状態など |
| ステップ | カード内に時系列順で並ぶ思考・感情・行動・身体反応などの要素 |
| 思考イベント | きっかけを起点に、思考・感情・行動・身体反応が連なって発生したひとまとまりの体験 |
| 振り返り | 記録済み内容を後から見返し、偏り・極端さ・抜けている視点を発見する行為 |
| マークダウン出力 | ChatGPTとの対話や外部保存に使うため、カード内容をMarkdown形式で出力する機能 |

## 4. 機能概要

### 4-1. 基本構造
ユーザーは、1カードを1つの思考イベントのまとまりとして登録する。  
各カードは共通項目として「きっかけ」を持ち、その配下に時系列順の複数ステップを登録できる。

### 4-2. カード内で扱う情報
カードでは以下を扱う。

- 作成日時
- カテゴリ（任意）
- きっかけ
  - きっかけ種別
  - きっかけ内容
- 時系列ステップ
  - 種別
  - 内容
  - 順序

### 4-3. ステップ種別
ステップ種別は以下とする。

- 思考
- 感情
- 行動
- 身体反応

### 4-4. 代表的な記録例
- きっかけ → 思考 → 感情 → 思考 → 行動
- きっかけ → 感情 → 身体反応 → 思考 → 行動
- きっかけ → 思考 → 思考 → 感情 → 行動

## 5. 要件定義

### 5-1. カード作成要件
- ユーザーは新規カードを作成できること
- 1カードは、ユーザーが1つのまとまりとして振り返りたい体験単位で登録できること
- カードには少なくとも以下の情報を保持できること
  - 作成日時
  - カテゴリ（任意）
  - きっかけ
  - 複数の時系列ステップ

### 5-1a. カテゴリ要件
- カードは任意の補助情報としてカテゴリを持てること
- カテゴリは、思考イベントの主構造とは独立したメタ情報として扱うこと
- カテゴリ未設定でもカードを作成・保存できること
- MVPでは、カテゴリは単一選択とすること
- カテゴリは、記録の文脈を大まかに見分けやすくする目的で使うこと

#### カテゴリ候補
初期要件として、以下の分類を持つこと。

- 仕事
- プライベート
- 学習

### 5-2. きっかけ要件
- カードは共通項目として「きっかけ」を持つこと
- MVPでは、主たるきっかけを1件保持すること
- きっかけには以下を保持できること
  - きっかけ種別
  - きっかけ内容

#### きっかけ種別
初期要件として、以下の分類を持つこと。

- 外部出来事
- 内部発生
- 身体状態

#### きっかけ種別の定義
| 種別 | 定義 | 例 |
|---|---|---|
| 外部出来事 | 外部から起きた事実や刺激を起点とするもの | 他人の発言、会議での出来事、Slackメッセージ、誰かの態度 |
| 内部発生 | 自分の内側で自然発生した思考や連想を起点とするもの | 急に不安になる、過去を思い出す、思いつきが出る |
| 身体状態 | 身体の状態変化を起点とするもの | 疲労、眠気、頭痛、空腹、動悸 |

### 5-3. ステップ要件
- カード内には複数のステップを保持できること
- ステップは表示順がそのまま時系列順となること
- 各ステップは以下の情報を持つこと
  - 種別
  - 内容
  - 順序

#### ステップ種別定義
| 種別 | 定義 |
|---|---|
| 思考 | 解釈、判断、連想、推測、自分への言葉、他者への見立てなど |
| 感情 | 怒り、不安、悲しみ、焦り、安心、嬉しさなどの感情 |
| 行動 | 発言、返信、黙る、席を外す、メモするなどの外部化された行為 |
| 身体反応 | 胸のざわつき、肩のこわばり、動悸、涙、胃の重さなど |

### 5-4. ステップ追加要件
- ユーザーはカード内にステップを追加できること
- MVPでは、追加されたステップは常に末尾に追加されること
- ステップ追加後、種別を選択し内容を入力できること
- ステップの削除ができること
- ステップの並び替え機能はMVP対象外とする
- 途中位置へのステップ挿入はMVP対象外とする

### 5-5. カード編集要件
- ユーザーは既存カードのカテゴリを編集できること
- ユーザーは既存カードのきっかけを編集できること
- ユーザーは既存カードの各ステップを編集できること
- ユーザーは既存カードのステップを削除できること

### 5-6. カード区切りの運用方針
- 1カードは、ユーザーが1つのまとまりとして振り返りたい出来事・体験単位を目安とする
- 同一テーマ・同一流れに属する内容は同一カードに含めてよい
- 場面や主題が大きく変わる場合は別カードとすることを推奨する
- 厳密な自動判定ルールは設けない

#### 同一カードにしてよい目安
- 同じ出来事の流れの中で起きている
- 1つの主要テーマに収まっている
- 後から見て「ひとまとまりの体験」として語れる

#### 別カードにした方がよい目安
- 場面が切り替わった
- 時間が大きく空いた
- 主題や感情の中心が大きく変わった
- 別の振り返り対象として扱いたい

### 5-7. マークダウン出力要件
- カード内容をマークダウン形式で出力できること
- 出力時には、少なくとも以下を含めること
  - 作成日時
  - カテゴリ（設定されている場合）
  - きっかけ種別
  - きっかけ内容
  - ステップの時系列順一覧
  - 各ステップの種別
  - 各ステップの内容
- 各ステップは、番号付きの時系列順を保ちつつ、「種別」と「内容」を分けて読める構造で出力すること

#### 出力例
```markdown
## 思考イベント
- 作成日時: 2026-03-22 10:30
- カテゴリ: 仕事

### きっかけ
- 種別: 外部出来事
- 内容: 会議で進め方に厳しい指摘を受けた

### ステップ
1.
- 種別: 思考
- 内容: 自分の進め方を否定された気がした

2.
- 種別: 感情
- 内容: イライラした

3.
- 種別: 身体反応
- 内容: 胸がざわついた

4.
- 種別: 思考
- 内容: また押し切られるかもしれないと思った

5.
- 種別: 行動
- 内容: 強めの口調で返した
```

## 6. UI要件

### 6-1. 共通方針
- 機能要件はスマホ・PCで共通とする
- UIはデバイス特性に応じて出し分ける
- 入力負荷を下げることを優先する
- 記録時に過度な整理や複雑な操作を強制しないこと
- 後から振り返りやすい視認性を確保すること

### 6-2. スマホUI要件
スマホでは、縦方向の流れと入力しやすさを優先する。

- 画面は縦積みレイアウトとする
- カード上部にカテゴリ選択UIを配置できること
- カード上部に「きっかけ」入力エリアを配置する
- その下に時系列ステップを縦に並べる
- ステップ追加は下部の「ステップを追加」操作で行う
- 各ステップは以下で構成する
  - 種別表示・変更UI
  - 内容入力欄
  - 最小限の削除操作
- キーボード表示中でも操作しやすいよう、完了や追加に関する操作をキーボード近傍または固定領域に配置できること
- テキスト入力欄は内容量に応じて自動で縦に伸長すること
- 一定以上の高さでは内部スクロールまたは最大高さ制御を行うこと

### 6-3. PC UI要件
PCでは、一覧性と編集効率を優先する。

- きっかけとステップ一覧を同一画面で広く表示できること
- カテゴリ、きっかけ、ステップの関係が一画面内で把握しやすいこと
- ステップはスマホより広い入力幅で表示できること
- 種別表示・変更UIは視認性高く配置すること
- マウス操作だけでなく、キーボード操作でも編集しやすいこと
- 必要に応じて、カードの左側に共通情報、右側にステップ編集領域などの2カラム構成を取れること
- PCではスマホより情報密度を高めてよい
- ただし、時系列の読みやすさを損なわないこと

### 6-4. ステップ追加UI要件
- ステップ追加操作は、原則として1つの追加導線に集約すること
- 初期案として、カード下部に「ステップを追加」操作を配置すること
- ステップ追加後に、種別選択と内容入力を行えること
- 種別は、各ステップ内で変更できること
- MVPでは、途中挿入よりも末尾追加を優先すること
- 見た目のノイズを増やしすぎないこと
- 種別ごとの追加ボタンを画面上に常時大量配置しないこと

## 7. デザイン要件

### 7-1. デザイン方針
本機能のデザインは、以下を満たすこと。

- クリーンで、情報整理がしやすいこと
- 入力に集中できること
- ダサく見えないこと
- ただし過度に派手でなく、振り返りや内省の邪魔をしないこと
- スマホ・PCいずれでも統一感を持つこと

#### デザインキーワード
- ミニマル
- クリーン
- モダン
- 落ち着き
- 視認性重視
- 情報の階層が明確

### 7-2. デザインシステム方針
- 現行の技術構成を前提とし、Tailwind CSSベースでデザインを組む
- アイコンは lucide-react を用いる
- コンポーネントライブラリを追加する場合でも、過度に雑多な見た目にならないよう統一トーンを維持する
- スマホ・PCでレイアウトは変えても、色・余白・角丸・文字階層のルールは共通化する

### 7-3. カラーパレット要件
#### 基本方針
- 背景は明るいニュートラルカラーを基本とする
- 主要な強調色は1系統に絞る
- 補助色は状態表現に限定し、無秩序に増やさない
- 彩度が高すぎる色の多用は避ける
- 長文入力時に目が疲れにくい配色にする

#### 推奨構成
- ベース背景: オフホワイト〜淡いグレー
- カード背景: 白または白に近い色
- 主アクセント: 青系または青緑系の落ち着いた色
- テキスト主色: 濃いグレー
- 境界線: 薄いグレー
- エラー・削除: 赤系
- 成功・保存済み: 緑系

#### ステップ種別カラー
ステップ種別は、種別判別のために軽い色分けを許容する。ただしベタ塗りではなく、チップやラベルで控えめに表現する。

例:
- 思考: 青系
- 感情: ピンク〜ローズ系
- 行動: グリーン系
- 身体反応: アンバー〜オレンジ系

※ 色は識別補助であり、主張しすぎないこと。背景全面を色で塗り分けないこと。

### 7-4. タイポグラフィ要件
- 文字サイズの階層を明確にすること
- 本文は長文入力を想定し、可読性重視のサイズと行間を確保すること
- ラベル、補助情報、本文、見出しでメリハリをつけること
- 英数字や日本語が混在しても崩れにくいこと
- 過度に細いフォントウェイトは避けること

#### 推奨階層
- 画面タイトル: 大きめ、強め
- セクション見出し: 中程度、太字
- ラベル: 小さめ、補助色
- 本文入力: 標準サイズ、十分な行間
- メタ情報: 小さめ、控えめ

### 7-5. 余白・レイアウト要件
- カード内外に十分な余白を取り、窮屈に見せないこと
- 要素同士はルールを持って間隔を統一すること
- スマホでは縦方向の呼吸感を重視すること
- PCでは横幅を活かしつつ、読み筋が散らないようにすること

#### 基本ルール
- セクション間余白 > 要素間余白 > ラベルと入力欄の余白、の順で階層化する
- カードの外周余白は、入力しやすさと見た目の軽さを両立する値にする
- ステップ同士の間隔は一定にし、時系列の並びが見やすいこと
- 余白不足による詰まり感を避ける
- 余白過多による間延びも避ける

### 7-6. 角丸・境界・影の要件
- カード、入力欄、ボタンは適度な角丸を持たせること
- 角丸は全体で統一すること
- 境界線は細く控えめにし、必要な場所に限定すること
- 影は強すぎないソフトシャドウを基本とすること
- 立体感は軽めに留め、重いUIにしないこと

#### 方針
- カード: やや大きめの角丸
- 入力欄: 中程度の角丸
- ボタン・チップ: 中〜やや大きめの角丸
- シャドウ: うっすら。カードの浮きすぎを避ける

### 7-7. アイコン要件
#### 利用ライブラリ
- アイコンは lucide-react を使用すること

#### 利用方針
- アイコンは意味補助として使用し、装飾過多にしないこと
- 種別ラベル、削除、保存、追加、完了など、意味が伝わりやすい箇所に限定して使うこと
- 同一画面でアイコンの線幅感やトーンを揃えること
- 塗りアイコンと線アイコンを混在させないこと

#### 想定利用箇所
- きっかけ
- 思考
- 感情
- 行動
- 身体反応
- ステップ追加
- 保存
- 完了
- 削除
- コピーまたはMarkdown出力

#### 留意点
- アイコンだけで意味を伝えようとしないこと
- ラベルやツールチップと併用すること
- スマホではタップ領域を十分に確保すること

### 7-8. 入力UIデザイン要件
- テキスト入力欄は、書くことに集中しやすいノイズの少ない見た目とする
- フォーカス時は、どこに入力しているかが明確に分かること
- エラー時、選択時、入力中などの状態変化が視認できること
- 入力欄の高さは内容に応じて自然に伸びること
- ステップ入力欄は、一覧として見ても種別と本文の対応が分かりやすいこと

### 7-9. ボタンデザイン要件
- 主ボタンと副ボタンの優先度が見た目で分かること
- ボタン数を必要最小限に抑えること
- 追加や完了など頻度の高い操作は押しやすいサイズにすること
- 削除など破壊的操作は主操作より目立たせすぎないこと
- スマホでは特にタップ領域を十分に確保すること

### 7-10. 状態表現要件
- 保存中、保存済み、未保存、編集対象、選択中などの状態が分かること
- 状態表現はテキスト・色・アイコンを適切に組み合わせること
- ユーザーが「今どういう状態か」を見失わないこと

### 7-11. スマホデザイン要件
- キーボード表示時の窮屈さを減らすこと
- 操作導線を下部寄りにまとめ、片手操作しやすくすること
- 連続入力しても画面がうるさく見えないこと
- テキスト入力と操作ボタンが干渉しないこと
- 縦長画面でも情報のまとまりが崩れないこと

### 7-12. PCデザイン要件
- 横幅を活かして情報を見渡しやすくすること
- 編集対象と全体構造を同時に把握しやすいこと
- スマホより高密度でも、読みにくさを生まないこと
- レイアウトを広げても間延びしないこと
- フォーム然としすぎず、カード体験として一貫させること

## 8. レスポンシブ方針
- データ構造、登録項目、出力形式はスマホ・PCで共通とする
- レイアウト、ボタン配置、入力欄サイズ、情報密度はデバイスごとに最適化する
- つまり「中身は同じ、見せ方を変える」方針とする

### 共通化するもの
- きっかけ
- カテゴリ
- ステップ種別
- 時系列構造
